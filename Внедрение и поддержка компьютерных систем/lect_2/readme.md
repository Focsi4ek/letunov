# Конспект 2. Виды тестирования

## 1. Классификация видов тестирования

### По знанию внутренностей системы:
* **Тестирование чёрного ящика (Black Box Testing / Closed Box / Opaque Box / Behavioral Testing)**:
    * тестировщик не имеет информации о внутреннем устройстве системы;
    * идеи для тестирования исходят из предполагаемого поведения пользователей.
* **Тестирование белого ящика (White Box Testing / Glass Box / Structural Testing)**:
    * основано на знании внутреннего устройства системы;
    * обычно выполняется программистами;
    * предполагает доступ к исходному коду и возможность написания кода, связанного с библиотеками тестируемого ПО (характерно для юнит‑тестирования).
* **Тестирование серого ящика (Grey Box Testing)**:
    * сочетает подходы чёрного и белого ящиков;
    * информированное тестирование, ориентированное на пользователя.

### По степени автоматизированности:
* **Ручное тестирование (Manual Testing)**.
* **Автоматизированное тестирование (Automated Testing)**.
* **Полуавтоматизированное тестирование (Semiautomated Testing)**.

### По субъекту тестирования:
* **Альфа‑тестирование (Alpha Testing)**:
    * имитация реальной работы с системой штатными разработчиками;
    * либо реальная работа с системой потенциальными пользователями/заказчиком;
    * проводится на ранней стадии разработки или как внутреннее приёмочное тестирование законченного продукта.
* **Бета‑тестирование (Beta Testing)**:
    * распространение ограниченной версии (по функциональности или времени работы) для группы лиц;
    * цель: убедиться, что продукт содержит мало ошибок, получить обратную связь от будущих пользователей.
* **Тестирование при приёмке (Smoke Testing)**.
* **Тестирование новых функциональностей (New Feature Testing)**.
* **Регрессионное тестирование (Regression Testing)**.
* **Тестирование при сдаче (Acceptance Testing)**.

### По времени проведения:
* на этапах создания (обследования);
* на этапах технического сопровождения;
* на этапах дальнейшего развития.

### По критерию «позитивности» сценариев:
* позитивные сценарии (проверка корректной работы при правильных входных данных);
* негативные сценарии (проверка обработки ошибок и некорректных данных).

### По степени изолированности тестируемых компонентов:
* **Модульное (компонентное) тестирование** — проверка отдельных частей системы.
* **Интеграционное тестирование** — выявление дефектов взаимодействия подсистем (используются компоненты, проверенные модульным тестированием).

### По степени подготовленности к тестированию:
* с подробной документацией и сценариями;
* без формализованной подготовки (исследовательское тестирование).

### По объёму тестирования:
* полное;
* выборочное.

## 2. По объекту тестирования

### Функциональное тестирование (Functional Testing)
* проверяет соответствие реализованных функций требованиям и ожиданиям пользователя;
* тестируется каждая функция приложения и их взаимодействие в комплексе;
* исследуются все сценарии использования.

**Виды функционального тестирования:**
* тестирование пользовательского интерфейса;
* тестирование локализации;
* тестирование безопасности;
* тестирование удобства использования;
* тестирование совместимости;
* тестирование инсталляции;
* тестирование документации.

### Нефункциональное тестирование (Non‑Functional Testing)
Оценивает характеристики ПО, которые могут быть измерены различными величинами («как» работает система).

**Виды нефункционального тестирования:**
* **Нагрузочное тестирование (Load Testing)** — проверка поведения системы при ожидаемой нагрузке.
* **Стрессовое тестирование (Stress Testing)** — оценка надёжности и устойчивости системы в условиях превышения пределов нормального функционирования (особенно важно для критически важного ПО).
* **Тестирование производительности (Performance Testing)** — измерение скорости и эффективности работы системы.
* **Объёмное тестирование (Volume Testing)** — проверка работы с большими объёмами данных.
* **Тестирование стабильности (Stability Testing)** — оценка способности системы работать без сбоев длительное время.
* **Тестирование на отказ и восстановление (Recovery Testing)** — проверка способности системы восстанавливаться после сбоев.
* **Тестирование безопасности (Security Testing)** — выявление уязвимостей и защита от несанкционированного доступа.
* **Тестирование удобства использования (Usability Testing)** — оценка простоты освоения и эффективности работы пользователя.
* **Тестирование локализации (Localization Testing)** — проверка адаптации приложения для пользователей из других стран.

## 3. Детализация отдельных видов тестирования

### Тестирование пользовательского интерфейса (UI Testing)
**Цель:** проверить:
* лёгкость освоения системы и документации;
* эффективность выполнения задач пользователем;
* защищённость системы от ошибок пользователя.

**Стандартные объекты UI и проверки:**
* Text box;
* Text area;
* Numeric field;
* Drop down list;
* Combo box;
* Check box;
* Date field.

### Тестирование локализации (Localization Testing)
Проверка адаптации приложения для пользователей из других стран:
* перевод текстов;
* корректность форматов даты, времени, чисел, валют;
* соответствие культурным особенностям.

### Тестирование скорости и надёжности
Оценка времени задержки в обработке как функции от приложенной нагрузки. Проводится для получения статистических и аналитических данных в условиях, приближённых к реальной эксплуатации.

**Включает:**
* нагрузочное тестирование;
* стрессовое тестирование;
* тестирование производительности;
* объёмное тестирование;
* тестирование стабильности.

---

## 4. Общие сведения

**Тестирование** — ряд мероприятий по испытанию объекта с целью:
* установления соответствия характеристик требованиям;
* выявления дефектов (ошибок в работе или неприемлемого качества функционирования).

**Результаты тестирования** — список выявленных несоответствий и дефектов.

**Методика испытаний системы** включает:
* перечень сценариев тестирования (test cases);
* подробное описание шагов;
* ожидаемые результаты для каждого шага;
* имитацию реальной эксплуатации системы.
